<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Schema → Database | Table definitions must enforce data integrity at DB level, not just application level |
| Application → Schema | Type inference from Drizzle must be accurate — no manual type overrides |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Tampering | audit_logs table | mitigate | No deleted_at column, no update path defined — audit entries are append-only per FR-11. Any mutation to audit_logs requires superadmin-level access (enforced in Phase 3 RBAC) |
| T-01-06 | Information Disclosure | users.password_hash | mitigate | Column named `password_hash` (not `password`) to signal hashed-only storage. bcrypt cost=12 enforced at application layer (Phase 2). No plaintext password column exists |
| T-01-07 | Repudiation | audit_logs.user_id | mitigate | user_id is nullable for system actions but required for user actions. Every user-initiated mutation MUST log the actor (enforced in audit plugin, Phase 10) |
| T-01-08 | Tampering | feed_stock quantities | mitigate | DECIMAL(10,3) prevents floating-point precision attacks. All stock changes go through feed_movements (append-only), not direct stock updates — enables full audit trail |
</threat_model>
