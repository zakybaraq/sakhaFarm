  <files>server/src/modules/rbac/rbac.service.ts, server/src/modules/rbac/rbac.errors.ts</files>
  <behavior>
    - createRole: inserts role with name, description, tenantId, isDefault=0 → returns created role
    - createRole: rejects if role name already exists for same tenant (ValidationError)
    - listRoles: returns all roles with userCount (COUNT from users table) and permissionCount (COUNT from role_permissions)
    - getRole: returns role by id with assigned permissions joined → throws NotFoundError if missing
    - updateRole: updates name/description, cannot update isDefault flag → throws NotFoundError if missing
    - deleteRole: throws RoleInUseError if any active users have this roleId, throws DefaultRoleError if isDefault=1, otherwise soft-deletes (or hard-delete if no users)
    - listPermissions: returns all permissions grouped by category
    - createPermission: inserts permission with name, description, category → rejects if name already exists
    - assignPermissions: for a given roleId, accepts array of {permissionId, action}, deletes existing assignments for that role, inserts new ones atomically in a transaction
    - All mutations call auditLog helper with actor info, action type, old/new values
  </behavior>

Create `server/src/modules/rbac/rbac.service.ts` with all functions listed above.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → RBAC API | Untrusted input for role/permission creation, must validate all fields |
| RBAC Service → Database | Trusted (Drizzle parameterized queries prevent SQL injection) |
| Session → Permission Guard | Session data (roleId) must be verified against database, not trusted blindly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | rbac.routes.ts | mitigate | Validate all request bodies with Elysia t.Object schemas — reject malformed role names, invalid permission IDs |
| T-03-02 | Elevation of Privilege | rbac.service.ts → assignPermissions | mitigate | requirePermission('rbac.update') guard on all role-permutation routes; Super Admin bypass only via roleId=1 check in requirePermission |
| T-03-03 | Repudiation | rbac.service.ts mutations | mitigate | Every RBAC mutation writes to audit_logs with userId, action, oldValue, newValue — cannot deny changes |
| T-03-04 | Information Disclosure | listRolesHandler | mitigate | Only return role data (no user passwords, no session tokens); role names and permission lists are non-sensitive |
| T-03-05 | Denial of Service | deleteRole endpoint | mitigate | Cannot delete default roles (isDefault check) or roles with active users (user count check) — prevents accidental lockout |
</threat_model>
